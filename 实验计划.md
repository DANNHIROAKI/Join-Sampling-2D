# 实验计划（2D，TH / Throughput 配置）

本计划与仓库根目录的 `run.sh` **严格对齐**：`run.sh` 的默认参数网格就是本文档中 A*~D* 四组实验。

> **TH（throughput）配置约定**
>
> - 固定：`d=2`，`THREADS=1`，`REPEATS=3`，`WRITE_SAMPLES=0`，`VERIFY=0`。
> - 输出的 wall/build/count/sample 计时来自 `apps/sjs_run` 的 phase 计时（写入 `phases_json`），后处理脚本会在 `summary_stats.csv` 中做 mean/median/cv 汇总，并对 C* 额外派生吞吐指标。
>
> **注意（重要）**：当前实现里即使 `WRITE_SAMPLES=0`，程序仍会在内存中保存 `t` 个 `PairId`（8 bytes/样本）。因此 **所有方法的 SAMPLE 阶段空间都是 O(t)**；`ours/enum_sampling` 额外还有 `O(|J|)` 的 join materialization 空间。

---

## 0. 方法（本仓库当前可跑的模型）

仓库当前的 Dim=2 baseline registry 只有 3 个（对应 `src/baselines/baseline_factory_2d.cpp`）：

1. **SJS-Sampling（Framework II）**：`--method=ours --variant=sampling`
2. **SJS-Enum（Framework I）**：`--method=ours --variant=enum_sampling`
3. **KDTree baseline（RS-over-SRJ）**：`--method=kd_tree --variant=sampling`

> 你文档里提到的“全 7 模型”如果来自更大版本的对照集合：请在后续把额外 baseline 接入 factory 后，再在 `run.sh` 的 `MODELS_MAIN=(...)` 里补上即可；本文档与当前仓库实现保持一致。

## 1. 数据集族（Alacarte-rectgen）

所有数据集均用 `tools/alacarte_rectgen_generate.py` 生成（`run.sh` 会复制并 patch 成 preset），并缓存为二进制关系文件（R/S 各一个 `.bin`）。

- **F0（参照族）**：`volume_dist=fixed`，`shape_sigma=0.0`
- **F1（更“苛刻/多样”族）**：`volume_dist=lognormal`，`volume_cv=1.0`，`shape_sigma=1.0`

### 1.1 参数解释与命名编码

- `N`：每侧关系大小（`nR=N, nS=N`）
- `alpha_out`：按 rectgen 定义的输出密度参数。
  - 实验中常用的粗略期望：
    \[
      |J| \approx \alpha_{\mathrm{out}}\cdot(|R|+|S|)=\alpha_{\mathrm{out}}\cdot 2N.
    \]

数据集名形如：

- `d2_F0_n1000000_a0p3_gs1` 表示 `d=2, fam=F0, N=1e6, alpha_out=0.3, gen_seed=1`
- 其中 `a0p3` 是把小数点 `.` 替换为 `p` 的安全编码。

## 2. 四组主实验（A*~D*）

### （A*）变化 \(\alpha_{\mathrm{out}}\)：密度敏感性（throughput 模式）

**目的**：把实验从“kd-tree 友好的小 \(t\)”切到“**SJS 强项的大 \(t\)**”，观察密度变化下的反转区间。

- 固定：`N=1,000,000`，`d=2`，`t=10,000,000`，TH 配置
- 扫描：
  \[
  \alpha_{\mathrm{out}}\in\{0.1,0.3,1,3,10,30,100,300,1000\}
  \]
- 数据族：F0 + F1（同一组 \(\alpha\) 各跑一遍）
- 跑的模型：3 个（SJS-Sampling / SJS-Enum / KDTree baseline）

> 备注：当 \(\alpha\) 很大时 `ours/enum_sampling` 可能因 `ENUM_CAP` 或内存不足被标为 Not runnable（这是预期行为）。

---

### （B*）变化 \(N\)：输入规模可扩展性（throughput 模式）

**目的**：在更大输入上考察随 \(N\) 的增长趋势；同时用 \(t=10^7\) 让吞吐优势更明显。

- 固定：`d=2`，`alpha_out=100`，`t=10,000,000`，TH 配置
- 扫描：
  \[
  N\in\{100{,}000,\ 200{,}000,\ 500{,}000,\ 1{,}000{,}000,\ 2{,}000{,}000,\ 5{,}000{,}000\}
  \]
- 数据族：F0（正文）
  - F1（可选）：设置 `INCLUDE_B_F1=1`（建议只跑部分点放附录）

**关于 enum_sampling 可运行性**

- `ours/enum_sampling` 需要 materialize \(|J|\) 个 pair，内存近似 `8*|J|` bytes。
- 例如 `N=5e6, alpha=100` 时，粗略 \(|J|\approx 1e9\)` → 仅 pair 数组约 8GB`，通常需要 32GB+ 内存才更稳。
- `run.sh` 默认 `ENUM_CAP=1,200,000,000`，你也可以按机器内存调大/调小。

---

### （C*）变化 \(t\)：输出规模可扩展性（SJS 主战场）

**目的**：直接刻画“吞吐量随 \(t\)”的相图，并让 ours/enum_sampling 的优势充分展现。

- 固定：`N=1,000,000`，`d=2`，`alpha_out=100`，TH 配置
- 扫描：
  \[
  t\in\{1{,}000{,}000,\ 3{,}000{,}000,\ 10{,}000{,}000,\ 30{,}000{,}000,\ 100{,}000{,}000,\ 300{,}000{,}000,\ 1{,}000{,}000{,}000\}
  \]

**额外派生指标（脚本自动生成）**

基于每次 run 的 `run_sample_ms`：

- 吞吐：`samples/sec = 1000 * t / run_sample_ms`
- 每样本开销：`ns/sample = 1e6 * run_sample_ms / t`

这些指标会写入：

- `run/temp/<EXP_TAG>/manifest/summary/summary_stats.csv`

---

### （D*）形状/尺度方差：数据族敏感性

**目的**：引入“更有利于 SJS 的数据族”，同时用 F0 做参照，避免“只挑对自己有利的数据”。

- 固定：`N=1,000,000`，`d=2`，`alpha_out=100`，`t=10,000,000`，TH 配置
- 两组数据族（对每个族跑全 3 模型）：
  1. **F0**：`fixed + shape_sigma=0.0`
  2. **F1**：`lognormal(cv=1.0) + shape_sigma=1.0`

（可选扩展）如果想系统扫描 `shape_sigma`：设置 `RUN_D_SHAPE_SWEEP=1`，并用 `D_SHAPE_SIGMAS="0.0 0.5 1.0 1.5 2.0"` 控制网格。

## 3. 运行方式与产物结构

### 3.1 一键运行

```bash
chmod +x run.sh
./run.sh
```

默认 `EXP_TAG=exp3_th_biggrid_2d`，产物会落在：

- 临时目录（可恢复/断点续跑）：`run/temp/<EXP_TAG>/...`
- 结果归档：`results/raw/<EXP_TAG>/...`
- 本次 run 的全量日志：`log/run_<EXP_TAG>_<timestamp>_pid<id>.log`

### 3.2 常用只跑子集

```bash
# 只跑 C*（t 扫描）
RUN_A=0 RUN_B=0 RUN_C=1 RUN_D=0 ./run.sh

# B* 同时跑 F1（附录用）
INCLUDE_B_F1=1 RUN_A=0 RUN_C=0 RUN_D=0 ./run.sh
```

### 3.3 大规模时的建议

- `t` 很大时（\(\ge 3\times 10^8\)），单次 run 的样本输出向量可能占用数 GB 内存；
  - 建议保持 `PAR_C_LARGE=1`（默认就是 1）并减少并行。
- 如果机器内存不足：优先把 `MAX_PARALLEL` 调小，或把 `TS_C` / `NS_B` 里最大的点拆出去单独跑。

---

## 4. 你最终会看到的关键文件

- `run/temp/<EXP_TAG>/manifest/merged/*_merged.csv`：每个 sweep 合并后的原始结果
- `run/temp/<EXP_TAG>/manifest/summary/summary_stats.csv`：汇总统计 + C* 吞吐派生指标
- `run/temp/<EXP_TAG>/manifest/FAILURES.tsv`：失败任务（应为空）
- `run/temp/<EXP_TAG>/manifest/NOT_RUNNABLE.tsv`：被 precheck/cap 判定不可运行的任务（通常主要是 enum_sampling 的极端点）
